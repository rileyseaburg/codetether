# Argo CD Deployment

This repo deploys CodeTether to Kubernetes through Argo CD using the Helm chart
at `chart/a2a-server`.

## Files

- `deploy/argocd/project.yaml`: Argo CD `AppProject` for CodeTether.
- `deploy/argocd/application.yaml`: Argo CD `Application` for the A2A server,
  including the production Helm overrides in `spec.source.helm.valuesObject`.
- `deploy/argocd/external-secrets-operator.yaml`: Argo CD `Application` for
  External Secrets Operator.
- `deploy/secrets/`: Vault-backed `SecretStore` and `ExternalSecret` resources
  that sync runtime secrets into `a2a-system`.
- `chart/a2a-server/examples/values-argocd-prod.yaml`: matching production values
  for local `helm lint` / `helm template` checks.
- `scripts/bootstrap-argocd.sh`: installs Argo CD if missing and applies the project/application manifests.

## Prerequisites

- Kubernetes context points at the target cluster.
- Argo CD can read this repo. The Application uses the SSH repo URL by default,
  matching the existing Argo CD GitHub repository configuration in this cluster.
  If the repo is private, add repository credentials first:

```bash
argocd repo add git@github.com:rileyseaburg/codetether.git \
  --ssh-private-key-path ~/.ssh/id_ed25519
```

- Runtime secret values exist in HashiCorp Vault KV v2. Kubernetes Secrets are
  generated by External Secrets Operator; do not hand-create them in
  `a2a-system`.

```bash
vault kv put secret/codetether/a2a-server/database \
  DATABASE_URL='postgresql://USER:PASSWORD@HOST:5432/a2a_server'

vault kv put secret/codetether/a2a-server/auth \
  A2A_AUTH_TOKENS='worker:TOKEN,admin:TOKEN'

DOCKER_CONFIG_JSON="$(kubectl create secret docker-registry gcp-artifact-registry \
  --dry-run=client -o json \
  --docker-server=us-central1-docker.pkg.dev \
  --docker-username=_json_key \
  --docker-password="$(cat /path/to/gcp-service-account.json)" \
  | jq -r '.data[".dockerconfigjson"]' | base64 -d)"

vault kv put secret/codetether/a2a-server/artifact-registry \
  .dockerconfigjson="$DOCKER_CONFIG_JSON"
```

- Vault Kubernetes auth has a role named
  `codetether-a2a-server-secret-sync` bound to service account
  `codetether-vault-sync` in namespace `a2a-system`. That role only needs read
  access to `secret/data/codetether/a2a-server/*`.

## Bootstrap

```bash
./scripts/bootstrap-argocd.sh
```

The Application is manual-sync by default so the cluster will not try to roll out
before the Vault-backed Kubernetes Secrets are synced.

Confirm the sync before deployment:

```bash
kubectl wait externalsecret -n a2a-system \
  a2a-server-database a2a-server-auth-tokens gcp-artifact-registry \
  --for=condition=Ready --timeout=180s

kubectl get secret a2a-server-database a2a-server-auth-tokens gcp-artifact-registry \
  -n a2a-system
```

After bootstrap, this is the expected pre-deploy state:

```bash
argocd app get codetether-a2a-server
# Sync Status:   OutOfSync
# Health Status: Missing
```

## Deploy

```bash
argocd app sync codetether-a2a-server
argocd app wait codetether-a2a-server --sync --health --timeout 600
```

## Verify

```bash
kubectl get pods -n a2a-system -l app.kubernetes.io/name=a2a-server
kubectl get ingress -n a2a-system
curl -fsS https://api.codetether.run/.well-known/agent-card.json
```

## Promoting New Images

Update the Application image tag, keep the local Helm values example in sync,
then apply the Argo CD manifests and sync:

```bash
yq -i '.spec.source.helm.valuesObject.image.tag = "1.4.4"' deploy/argocd/application.yaml
yq -i '.image.tag = "1.4.4"' chart/a2a-server/examples/values-argocd-prod.yaml
git commit -am 'Deploy CodeTether 1.4.4 via Argo CD'
git push
kubectl apply -k deploy/argocd
argocd app sync codetether-a2a-server
```

For a forced rollout with the same image tag, change:

```yaml
blueGreen:
  rolloutId:
    green: "2026-05-06-01"
```

## Automated Sync

After the first successful manual sync, enable automated sync if desired:

```bash
argocd app set codetether-a2a-server --sync-policy automated --auto-prune --self-heal
```
